## 一、根因结论（三份报告已交叉验证，确认成立）

当前系统的核心缺陷不是 7 个互相独立的 bug，而是 **窗口状态被分散维护在多个无协调的数据源中**，外加 **disconnect handler 缺乏身份校验**，二者叠加产生了一系列表面看起来不相关的现象（假离线、假在线、Busy 错乱、初始化状态卡住）。

### 1.1 直接根因：disconnect handler 缺乏引用比较

`registerDisconnectHandler`（BrowserPool.ts#L325-L368）的回调闭包捕获了 `conn` 对象，但所有清理操作（L338 / L362 / L363）都只用 `windowId` 字符串做 key，**没有判断"此刻要清理的连接，是否还是触发本次回调时对应的那个连接"**。

时序证据（已用代码行号追踪确认）：

```
旧连接 disconnected 事件 → 回调进入异步队列
                ↓（几乎同时）
EasyBR 自动重启窗口 → refreshConnectionStatus 检测到
                ↓
connectAndSetupWindow 建立新连接 → connections.set(windowId, 新conn)
                ↓
P0验证通过 → p0Verified.add(windowId)
                ↓（旧回调此时才真正执行）
旧 disconnect 回调继续跑 → p0Verified.delete(windowId)
                ↓
结果：连接其实是好的，但状态被刚执行的旧回调误删
```

这就是"窗口已成功打开并导航到首页，但前端显示离线"的根本原因。

### 1.2 根因的连带破坏范围（比最初判断的更广）

代码级复核发现，这个缺陷不只影响 `p0Verified` / `windowBusy` / `connections` 三个集合，还连带破坏了两个独立子系统：

- **WindowLockManager 被偷锁**：disconnect handler 调用 `release(windowId)` 时**不带 taskId**，会无条件删除锁表条目（WindowLockManager.ts#L111-116）。在竞态场景中，如果新任务 B 此刻已经持有了同一 `windowId` 的锁，旧的过期 disconnect 回调会把任务 B 的锁直接偷走，而任务 B 自己后续的 `release(windowId, taskId_B)` 因为身份校验不匹配会变成空操作——**锁已经丢了，任务 B 却毫无察觉**。
- **SessionManager 心跳被误停**：同样的空 catch 块包裹着 `stopHeartbeat()` 调用（BrowserPool.ts#L345-L350），过期回调触发时会把刚启动不久的新心跳一并停掉。

**重要结论：引用比较守卫如果加在清理逻辑的最前面，能一次性覆盖 connections / p0Verified / windowBusy / WindowLockManager.release / SessionManager.stopHeartbeat 这五处清理动作**，因为它们都在同一个 disconnect handler 回调体内，被同一个守卫语句拦截。这意味着这一个修复的实际收益，比最初评估的"解决假离线"要大得多。

### 1.3 三个状态集合彼此无一致性校验（已逐一核实读写位置）

| 集合 | 写入位置 | 删除位置 | 跨集合校验 |
|---|---|---|---|
| `connections` | connectAndSetupWindow, refreshConnectionStatus, toggleWindow | disconnect handler, 多处 reset | 无 |
| `p0Verified` | doConnectAndSetup（P0三轮检查通过后） | 仅 disconnect handler | 无 |
| `windowBusy` | AssignmentEngine（获取窗口锁后） | AssignmentEngine（释放后）+ disconnect handler | 无 |

没有任何统一的 setter 或状态机来保证这三者步调一致，这是"多状态源问题"的代码层证据。

## 二、独立确认的额外问题（不属于竞态根因，但同样需要处理）

### 2.1 任务分配数据源不一致（新发现，优先级高）

`AssignmentEngine.selectOnlineWorker` 选人时读的是**数据库**里的 `is_connected` 字段；而真正拿连接执行任务的 `getStaffConnection` 读的是 **BrowserPool 内存中的 `connections` Map**。这是两个完全独立的数据源，且数据库字段的更新本身也依赖 disconnect handler 写入，同样受竞态影响。

后果：任务可能被分配给一个数据库里显示"已连接"、但内存里早已不存在或已被新连接替换的"幽灵窗口"，表现为**任务提交成功，但立刻因为找不到连接而失败**。

### 2.2 任务执行前不做同步复验

`ensureWindowReady`（BrowserPool.ts#L855-874）只做清标签页、清弹窗、展开侧边栏，**不重新验证 P0 状态**。任务分配引擎完全信任轮询缓存里的状态，而这套缓存本身存在最长 30 秒的更新延迟（见 2.3）。

### 2.3 GET 接口与服务端刷新频率不匹配

前端每 5 秒轮询 `GET /api/sites/:siteId/windows`，但该接口本身不触发刷新，只读缓存；服务端每 30 秒才主动刷新一次。窗口恢复后最长有 30 秒的状态延迟窗口期。

**注意（已修正一个早期误判）**：不应在 GET 接口内部直接触发 reconnect，因为这会把重连频率和 5 秒轮询频率绑死，对 EasyBR 造成不必要的高频压力。正确做法是新增一个独立的后台巡检任务。

### 2.4 EasyBR 不可达时无法区分"真离线"与"状态源异常"

`openedList` 返回空数组时，所有窗口会被判定为 offline，没有任何日志区分这是"窗口真的离线"还是"EasyBR 服务本身连不上"。代码级复核确认 EasyBRClient 已有基础熔断和健康检查缓存，缺的只是结果到前端的语义区分，不是熔断本身。

### 2.5 重连路径心跳孤儿问题（新发现）

`startHeartbeat` 只在服务启动阶段的 `initialize()` 完成后被调用一次。**两条重连路径（`refreshConnectionStatus` 和 `toggleWindow`）建立新连接后都没有重新调用 `startHeartbeat`**。

后果：任何经历过自动重连或手动 toggle 复活的窗口，从此永远没有心跳保活。Session 会在约 30 分钟后静默过期，系统不会主动发现，只能等下一次任务请求返回 302/401 才被动暴露——表现为"窗口显示 ready，任务执行到一半却登录失效"。

好消息：`startHeartbeat` 自身已有防重入保护（SessionManager.ts#L302-305），补调用是无副作用的安全改动。

### 2.6 init_window 任务路径绕过了标准锁

`AssignmentEngine.ts#L380-454` 中 `init_window` 类型的任务完全跳过 `WindowLockManager`，只靠 `markWindowBusy` 软标记互斥。这意味着两个 `init_window` 任务并发操作同一窗口时不会被真正互斥。

**这一点需要特别向到派一体任务分配板块的开发提出明确要求：新功能里所有任务类型必须统一走 WindowLockManager 的标准 acquire/release 流程，不允许像 init_window 一样开特例**，否则这次修复堵上的洞会在新功能里以另一种形式重新打开。

### 2.7 异常被静默吞掉，无法被发现

disconnect handler 中 `stopHeartbeat` 和 `WindowLockManager.release` 调用包在空的 try-catch 里，连 warn 日志都没有。如果这两个调用因为意外原因失败（比如内部 Map 被并发修改），心跳泄漏和锁泄漏都不会留下任何痕迹。

## 三、本报告否定或修正的早期判断

为避免实现时产生歧义，明确记录三轮评估过程中被修正的判断：

1. **"GET 接口内部触发 reconnect"** —— 已否定。会和前端轮询频率绑死，造成不必要的高频重连压力。正确做法是独立的后台巡检任务。
2. **"需要重新设计心跳机制"** —— 部分否定。心跳机制（SessionManager）已经存在，问题是重连路径没有调用它，而不是机制本身缺失。优先修复调用缺口，而不是重新设计。
3. **"需要新建任务执行锁"** —— 已否定。WindowLockManager 本身就是为任务互斥设计的真实锁，普通任务路径已经在使用。问题是 disconnect handler 在特定竞态下会绕过身份校验偷走这把锁，不是锁本身缺失。
4. **"generation 计数器 vs 闭包引用比较"** —— 两者对解决当前这一个竞态问题是等价的，引用比较改动量更小（约 5 行），可作为立即上线的热修复；generation 机制更适合作为未来统一状态机的版本控制基础设施，二者不是互斥关系，是不同阶段的工具。

## 五、给到派一体任务分配板块的明确要求

这套窗口状态监控机制的稳定性，是任务分配板块能否正确运行的前提，因此在该板块开发过程中必须遵守：

1. **不允许引入新的状态读取路径**——所有"窗口是否可用"的判断必须经过 Phase 2-1 统一后的 `WindowState`，不要再从 DB 或其他缓存读取一份独立的判断依据。
2. **不允许跳过 WindowLockManager**——所有任务类型（包括未来新增的"到派一体"相关任务）必须走标准 acquire/release 流程，参考 2.6 中 init_window 的反面案例。
3. **任务分配前必须同步复验**——不要假设轮询展示的状态在任务真正执行那一刻仍然成立，尤其是在用户手动勾选多个窗口批量执行任务的场景下，窗口数量越多，状态过期的概率越高。

## 六、结论

七个表面问题、两份独立评估、一次代码级复核，最终收敛到同一个结论：**问题不在于某一处代码写错了，而在于状态散落在太多地方，且唯一的清理入口（disconnect handler）缺乏身份校验**。好消息是，最根本的那个修复改动量极小（约 5 行），且这一行代码的收益范围比最初预想的要大——它同时保护了连接状态、P0验证、Busy标记、任务锁、心跳这五套机制。

建议执行顺序：先上 Phase 0 止血，验证窗口状态不再出现假离线/假在线后，再按 Phase 1 → Phase 2 的节奏推进，Phase 2 的统一状态机重构与到派一体任务分配板块的开发应该合并排期，避免两边各自摸索出不一致的状态判断逻辑。
